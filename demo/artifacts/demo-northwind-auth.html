northwind-sync · design only, no code

One decision gates every line of code.

How does northwind-sync hold credentials for each customer tenant? The handoff is 340 lines and there is no code yet — because nothing can be written until this is answered. The credential shape threads through provisioning, the credential store, token exchange, the executor, and revocation. Pick the shape and the modules fall out; guess it and you rewrite them.

Why this one blocks the rest

northwind-sync is a multi-tenant integration service: one deployment acts on behalf of many customers' tenants, unattended. Unattended is precisely what an interactive sign-in cannot do — so the service holds a machine credential per tenant, and the type of that credential is not a detail. It decides the columns in the credential table, whether there is a provisioning flow at all, how a token is minted, and what "revoke customer X" even means. Every module downstream inherits its shape.

There are exactly three doors. Two of them build something real now; the third builds against a mock and pays later. The trap is that the third feels like the cheap one.

The three doors

A — a single delegated-admin consent grant. One app registration, one code path. Each customer's admin grants it broad, org-wide consent and you are live this week. The cost is baked into that sentence: consent is org-wide, and revocation is all-or-nothing — a customer offboards by yanking the whole grant, and you cannot scope what the service can reach inside their tenant.

B — a per-tenant app registration with certificate credentials. Each tenant gets its own registration and its own cert; the service authenticates as that tenant only. It needs a real provisioning flow and certificate rotation, so it is the slow door. In exchange it is least-privilege by construction, revocation is per-tenant, and it is the shape an enterprise security review will actually pass.

C — defer, and build against a mock tenant. Momentum stays high because you write features instead of plumbing. But an undecided credential shape does not stay contained — it leaks into every module as a placeholder that each new file has to guess around. Deferral is not free; its cost compounds with every module you add before deciding.

A / B / C across the four axes that matter
Asingle consent Bper-tenant cert Cdefer / mock
Build time days — one app reg ~2 weeks — provisioning + rotation hours now, compounds
Blast radius org-wide consent, per customer per tenant, contained leaks into every module
Revocation all-or-nothing per tenant undefined
Security review conditional / likely fails passes nothing to review yet

The recommendation: B

Build B. Least-privilege and per-tenant revocation are not polish — they are the two things a customer's security team checks first, and A cannot offer either. The honest comparison is not "fast A versus slow B"; it is slow-now versus slow-later-plus-a-rewrite. Retrofitting B onto A or C means reopening the credential store and every module that ever touched a credential — which is the exact cost C claims to defer, moved to a worse week.

There is one move that is right under all three doors, and it makes a wrong call cheap: put the credential store behind an interface now, so modules depend on the abstraction rather than the auth shape. Do that first and even choosing A or C buys you an escape hatch instead of a rewrite.

Modules: PRV provisioning · CRD credential store · TOK token exchange · EXE executor · REV revocation. Deferring (C) does not shrink the blast radius — it makes every module carry an undecided credential shape.
