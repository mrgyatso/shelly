The index that stopped being used.

p99 · claim_job overnight · throughput flat
12ms → 840ms 70× slower

Overnight, p99 on claim_job went from 12 ms to 840 ms with throughput dead flat — so it wasn't load, it was a plan change. Bisecting the migrations, the regression lands exactly on 0042_add_tenant_id: it rebuilt the hot composite index leading with a column the query never filters on, and Postgres quietly fell back to a sequential scan over 4.2M rows.

Ruling out load first

The tempting story is traffic, so I killed it first: requests/sec on claim_job was within 3% of the prior week and CPU sat flat through the jump. Latency climbed while volume didn't — that's per-query cost, not contention. Which meant the change was in the plan, and the only thing that changed plans overnight was a migration.

Bisecting the migrations

Six migrations landed in the nightly deploy. Replaying them one at a time against a snapshot, the claim path holds at ~12 ms right up to 0042_add_tenant_id.sql — and from 0042 forward every claim takes ~840 ms. 0042 added a tenant_id column and, in the same file, rebuilt the claim index as (tenant_id, status).

Why the planner gave up on the index

Every hot query on this path filters (status, run_after) and never mentions tenant_id. A B-tree is only usable from its leading column inward — with tenant_id first and left unconstrained, the index can't satisfy the predicate at all, so the planner discards it and sequential-scans the whole table for a single claim: 

# EXPLAIN (ANALYZE, BUFFERS) — the claim path, after 0042
 Limit  (cost=0.00..91442.00 rows=1 width=284) (actual time=811.9..811.9 rows=1 loops=1)
   ->  Seq Scan on jobs  (cost=0.00..91442.00 rows=1 width=284)
         Filter: (status = 'pending' AND run_after <= now())
         Rows Removed by Filter: 4238119
         Buffers: shared hit=1216 read=57104
 Planning Time:  0.118 ms
 Execution Time: 840.213 ms

The rows=1 estimate is right — exactly one job gets claimed. But it pays a full-table filter to find it: 4,238,119 rows removed per claim. That filter is the 840 ms. Here is the index before and after 0042: 

-- before 0042  ·  ~12 ms, ordered index scan
CREATE INDEX jobs_claim_idx ON jobs (status, run_after);

-- after 0042  ·  leading column is never filtered →
-- claim seq-scans 4.2M rows  ·  ~840 ms
CREATE INDEX jobs_claim_idx ON jobs (tenant_id, status);

The fix: a partial index

The claim query only ever wants status = 'pending' rows, and pending is under 1% of the table. A partial index on (status, run_after) WHERE status = 'pending' restores the ordered lookup and — because it indexes so few rows — lands ~40× smaller than the old full index. CONCURRENTLY builds it without locking writes, so it ships online with no downtime: 

-- the fix: partial index on the real predicate, built online
CREATE INDEX CONCURRENTLY jobs_pending_claim_idx
    ON jobs (status, run_after)
    WHERE status = 'pending';

-- pending < 1% of rows → ~40× smaller than jobs_claim_idx
  • Before 0042 — p99 12 ms, ordered index scan, ~40 buffers touched per claim.
  • After 0042 — p99 840 ms, seq scan, 4.24M rows filtered for every single claim.
  • Partial index — indexes <1% of rows, ~40× smaller, builds CONCURRENTLY with no write lock.
  • Still open — whether (tenant_id, status) should keep tenant_id leading at all, or be reordered.
